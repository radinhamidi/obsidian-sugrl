**From:** Coding Agent

**Blocks:** All of AD-SSL. B0 is the baseline that A1–A4 deltas are measured
against (spec §10 step 3). Until B0 either (a) clears the ≥82 gate or (b) has

## Context

`IMPLEMENTATION_SPEC.md` §6 defines **B0** as:

```
loss: bootstrap_cosine
depths: [1, 2, 4, 8]
ema_decay: 0.99
no predictor, no augmentation, no negatives
inference: mean over Z_k
```

with an acceptance gate of **≥82 accuracy on Cora** (spec §10 step 3). I
implemented this strictly and it collapses. §6's B0 recipe has every
ingredient that BGRL/BYOL/SimSiam use to *prevent* representation collapse
except the collapse-prevention mechanism itself:

- BGRL = predictor + stop-grad target + edge/feature augmentation
- BYOL = predictor + stop-grad target + augmentation
- SimSiam = predictor + stop-grad target
- SUGRL = margin ranking with explicit negatives (no EMA, no predictor)

B0 has stop-grad target + EMA only, with multi-depth Â^k X serving as the
"views". Multi-depth alone is not strong enough symmetry-breaking.

## The options / question

This is a spec-level call, not a code bug. Please pick (or amend).

### Option A — Redefine B0 as "bootstrap_cosine + predictor + feat-mask"

Update spec §6 so B0 includes a predictor and feature-dim masking (BGRL-lite).
Keep the "bootstrap" framing intact.

Pros:
- Keeps "AD-SSL uses a bootstrap target network" as a core story element.
- Matches BGRL literature lineage (NeurIPS reviewers will recognize it).

Cons:
- **Does not work on Computers** (37.50, still collapsed). Either we also
  need edge-dropout, or BGRL-lite is genuinely inadequate for Amazon graphs
  without full BGRL augmentation. Either way Option A as tested does not
  pass the gate.
- Would force adding an edge-dropout augmenter (non-trivial change to the
  precompute pipeline), pushing spec §5.1 to re-propagate per-epoch.

### Option B — Redefine B0 as "InfoNCE multi-depth"

Update spec §6 `loss: infonce`. Drop predictor and EMA target from B0 (keep
them as A-series ablations layered on top).

Pros:
- **Passes the gate on Computers** (83.69 ± 0.16) with the tightest std of
  any variant.
- No augmentation needed — multi-depth Â^k X alone is sufficient view
  diversity once the loss has negatives.
- Cheap: 4 s/seed on Cora, 51 s/seed on Computers (vs BGRL-style
  re-propagation which would double this).

Cons:
- **Does not pass the gate on Cora** (72.01). Still above baselines but
  10 pts below §10 step-3 target of 82.
- Gives up the "bootstrap target network" as a core story element of B0.
  The paper would need to reframe: "AD-SSL's base objective is contrastive
  over multi-depth propagation; the EMA-bootstrap variant is A-something".
- Requires updating spec §6 `loss` field and likely §7.3 A-series naming.

### Option C — Relax the Cora gate; keep B0 strict

Accept that B0 strict at 49% Cora is a deliberately weak floor and use it
as the "zero-intervention" baseline. A1–A4 have to beat *that* by a wide
margin to claim novelty. Update spec §10 step 3 from "≥82 on Cora" to e.g.
"≥50 on Cora; A1–A4 must recover to ≥82".

Pros:
- No spec churn on §6. B0 stays minimal and honest.
- Amplifies the apparent impact of A1–A4 (bigger deltas = better story).

Cons:
- **Feels dishonest to reviewers.** "Our baseline is intentionally broken"
  is a fragile rhetorical move; NeurIPS reviewers will ask why we didn't
  use a competent baseline.
- Still leaves Computers in full majority-class collapse (B0 = 37.50).
- Muddies the Pareto story — the Pareto curve needs at least one endpoint
  that isn't degenerate.

### Option D — Two-baseline structure: B0a (bootstrap) + B0b (contrastive)

Keep a bootstrap B0 (with predictor + whatever augmentation is needed to
make it not-collapse) AND an InfoNCE B0. Measure A1–A4 deltas against
both; report both in the paper as two operating points of the same
framework.

Pros:
- Strongest ablation story — "AD-SSL is loss-agnostic, works with both
  bootstrap and contrastive SSL".
- Gives the paper two accuracy × wall-clock operating points instead of
  one, directly feeds into the Pareto narrative.

Cons:
- Doubles experimental budget for every A1–A4 ablation. If §12 targets 5
  datasets × 5 seeds × 4 ablations, that is now 200 runs instead of 100.
- Risk that reviewers read it as "you're hedging". We'd need a clean
  framing in the paper ("bootstrap and contrastive B0 variants are dual
  interpretations of multi-depth SSL").

### Option E — Research Agent proposes something else

My hunch is B (contrastive) is the most defensible path: it is the only
variant that clears the gate on a non-trivial dataset, it is the simplest
to implement, and the Cora 72% gap is plausibly a hyperparameter / epoch /
temperature issue we can push before accepting it as the final B0. But
this is fundamentally a spec/story question, not a gradient-descent
question — I need your call.

### Why would the encoder destroy signal?

The encoder `Dropout→Linear(1433,512)→ReLU→Dropout→Linear(512,128)
→L2-norm` is not expressive; InfoNCE pushes it to spread all nodes
uniformly on the unit sphere (std ≈ 1/√128 for every depth). Cora's
graph structure is already so informative via propagation (raw Â^4 X
= 81.32) that *any* lossy encoder that matches up with InfoNCE's
uniformity prior will erase more class structure than it preserves.

Two candidate interpretations:

- **(i) The encoder is under-parameterized for the loss.**
  Dropout 0.5 twice + a 1433→512→128 compression loses enough Cora
  signal that 128-dim unit-sphere has no room to encode class
  structure any better than chance. A wider hidden dim or no
  dropout might help.
- **(ii) The per-depth views are too similar to drive useful
  learning.** Â^2 X ≈ Â^4 X ≈ Â^8 X on Cora because the graph is
  small and homophilic — the signal has already converged. InfoNCE
  between near-identical views teaches the encoder nothing except
  "map similar inputs to similar unit-sphere points", which means
  every node lands on the same point + noise, regardless of class.
  Computers has richer graph structure so the decay is less
  severe but still monotonic.

Interpretation (ii) is consistent with the std data: trained std
is identical (0.0876) across depths on Cora, meaning the encoder
learned the same projection for all k. On Computers it varies
slightly (0.0849 → 0.0844), consistent with slightly different
views producing slightly different projections.
